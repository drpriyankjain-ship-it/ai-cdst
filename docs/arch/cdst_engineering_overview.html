System Architecture

Session Flow

One WebSocket per consultation. Three button presses. Three pipeline stages fire at precise moments.

Mobile App (RN)

Nurse presses Start. Streams base64 Opus audio chunks. Sends 3 marker events. Receives streamed tokens + stage_complete payloads.

⟶

Orchestrator

FastAPI WebSocket server. Owns full session lifecycle. Manages Vault, Deepgram connection, ring buffer for reconnect, and asyncio task dispatch.

⟶

Deepgram STT

Streaming WebSocket. Returns real-time interim + final transcripts. Finals appended to Vault incrementally (no full rewrites).

⟶

Postgres + pgvector

sessions (Vault JSONB), patient_records, stg_chunks (384-dim embeddings), confirmed_encounters. IVFFlat index for ANN search.

▲ Marker A — ~30s

  • Name, age, village, chief complaint captured
  • History Stage fires
  • Questionnaire first token in <1.5s
  • Two async tasks: stream + structured JSON

▲ Marker B — ~3-4 min

  • Structured interview complete
  • Diagnosis Stage fires
  • Differential streams in <1.5s
  • Clarifying questions generated after DDx

▲ Marker C — ~5-6 min

  • Clarifying phase complete
  • Management Stage fires
  • Full pipeline ~7-8s incl. rule engine
  • Triage decision + doctor handoff

RECONNECT RESILIENCE

Server keeps a 60s ring buffer (600-element deque). On reconnect, SessionState.from_vault() rebuilds state from Postgres. Client flushes buffered audio to a fresh Deepgram connection. No audio is lost.

Pipeline Detail

The Three Stages

Each stage is a fixed sequential pipeline with bounded LLM calls — no loops, no agents yet.

History Stage
2 calls · no RAG · target: <1.5s first token
Call 1 ~700ms · Extract chief complaint from 30s transcript. Schema has many null fields by design — only complaint + duration known at this point.
Call 2 ~1.3s · streaming · Build contextualised questionnaire. New patient = full intake (PMH, FH, SH, meds, allergies). Returning patient = short verification pass for changed fields only. Outputs patient_record_fields schema for Diagnosis Stage.
Two async tasks fire simultaneously: streaming questionnaire to nurse + structured JSON write to Vault.
Diagnosis Stage
3 calls · no RAG · full pipeline ~5-6s
Call 1 ~900ms · Extract structured medical concepts. Negatives captured explicitly. Ambiguous answers → uncertain_findings (not collapsed into positives).
Call 2 ~3.2s · streaming · Ranked differential (4-6 conditions). Two-layer epi prior: Layer 1 = common WB presentations string; Layer 2 = district + season lookup from epi_prior_wb.json. 34 must-not-miss diagnoses flagged from JSON regardless of rank. 11-field canonical schema, always validated.
Call 3 ~1.4s · Gap analysis + clarifying questions constrained to bedside_tools.json. uncertain_findings surfaced as priority re-asks.
Management Stage
4 calls · RAG in Call 2 · full pipeline ~7-8s
Call 1 + RAG parallel · ~900ms · Extract clarifying findings. RAG retrieves STG chunks for all DDx diagnoses simultaneously (top-8 per diagnosis, max 16 chunks).
Call 2 ~2.5s · streaming · Problem list: all clinical issues (acute / established / incidental / deferred). Every Rx item carries for_problem attribution. STG citation required per drug. Formulary JSON injected directly.
Call 3 ~1.8s · 5-dimension risk assessment: diagnostic uncertainty, iatrogenic risk, delay risk, complication watch, mitigation plan.
Call 4 ~1.2s · Triage decision + patient instructions. prescription_issued built in Python from Call 2 output — LLM never re-derives the drug list.
Deterministic rule engine runs after Call 4. Can only escalate LOW→HIGH, never downgrade. Config-driven via escalation_rules.json — MO-maintained without code deploys.
Design Decisions

Key Architecture Principles

Deliberate choices — not defaults. Each has a specific patient safety or operational reason.

🔌

Continuous WebSocket, not file uploads

Button presses are timestamp markers on a live stream. Eliminates STT startup latency for Stages 2 and 3. The 1.5s first-token target is only achievable this way.

🚫

No RAG in History or Diagnosis

LLM general clinical knowledge is reliable for differential generation. RAG belongs only in Management where it governs drug selection — a hallucinated dose causes patient harm.

⚙️

Deterministic rule engine for safety decisions

Rules are versioned, MO-maintained, auditable, and deploy independently via escalation_rules.json. Can only escalate LOW→HIGH. LLM never has final safety authority.

🔒

Prescription built in Python, not LLM relay

prescription_issued is assembled from Call 2's problem_list in Python. LLM cannot paraphrase, reorder, or drop drugs when building the handoff package.

🏗️

Problem-oriented schema, not single-diagnosis

Management Stage outputs a problem_list covering all clinical issues. Enables correct attribution per drug, handles multi-morbidity, and matches how clinicians actually think.

📦

Layer 3 epi prior gated by 3-step confirmation

Provisional diagnoses never write back. Gate 1: RDT result. Gate 2: treatment response. Gate 3: doctor agreement. Confidence weight 0.5 → 1.0. Prevents hallucinated data poisoning the prior.

📋

Vault is append-only after session close

The JSONB session document is the immutable clinical audit trail. Stages write incrementally during the session. Never modified after close — medicolegally important.

🛡️

Formulary as JSON, not vector store

The essential medicines list is small, structured, and clinic-specific. Direct prompt injection is faster and more controllable than retrieval. Per-clinic stock flags verified at onboarding.

Blockers & Open Questions

What We Need Answered

Three questions block downstream build. Two are product/clinical decisions — one is an STG ingest execution task.

Product / Clinical Lead
Patient Identity

How does the nurse look up or create a patient at session start?

Search by name / village? Physical patient ID card? Auto-create with later dedup? Blocks orchestrator patient-load logic and the mobile registration flow.

Standing Orders — Most Consequential Clinical Question

Can the nurse dispense before the doctor reviews (LOW risk)?

If yes → nurse operates under standing orders and the triage output is framed as a treatment plan. If no → output is framed as a draft pending auth. Determines the entire authorization flow design and the doctor-review interface.

Post-Management Nurse UI

Can the nurse add notes or flag disagreement before the case goes to the doctor?

Affects Management Stage output schema and what fields the doctor review interface must surface.

Engineering Execution
STG Corpus — Blocking RAG in Management Stage

No STG documents have been ingested into pgvector yet

Priority order: NHM STG all volumes → NVBDCP malaria ACT protocol (CRITICAL — Rx depends on this) → NHM kala-azar guidelines → RNTCP/NTP TB guidelines. Run scripts/ingest_stg.py against each. Similarity threshold 0.55, top-8 per diagnosis.

Per-Clinic Formulary Verification

formulary_wb.json reflects MoHFW Annexures — not real stock

Before any pilot site goes live, a Medical Officer must verify actual SHC-HWC stock against the formulary. The MO_REVIEW_CHECKLIST.md covers this step.

Multi-Complaint Differential (Known Gap)

Diagnosis Stage generates one DDx for the primary complaint only

Multi-morbidity patients with two simultaneous acute complaints are handled by the problem_list in Management, but Diagnosis Stage DDx only covers the primary. Logged in ADR 003. Low priority for pilot but needs resolution before scale.
