## What the Model Rollback Fixed

The 3.x preview models (`gemini-3-flash-preview`, `gemini-3.1-flash-lite-preview`) were 3–9× slower than their 2.5 equivalents, even without explicit `thinking_config`. D2 dropped from 22.4s to 6.7s; M2 from 22.0s to 6.5s.

`gemini-3.1-flash-lite-preview` was also triggering 503 errors in run_004 (M1, M4 cascaded to 2.5-flash-lite). No such errors in run_005 with 2.5-flash-lite as primary.

## What Thinking Suppression Fixed

On first run with 2.5-flash models (before adding thinking_budget=0), D2 took 32.6s (vs 6.7s in run_002) and D3 produced a truncated response. This confirmed that `gemini-2.5-flash` now has thinking **enabled by default**, unlike the version used in run_002. `thinking_budget=0` restores the non-thinking speed: D2 6.7s (identical to run_002), H2 10.1s (vs 18.0s run_002 — additional improvement beyond run_002, possibly from smaller/faster response schema).

Thinking suppression added to: H2, D2, D3, M2, M3.
Not added to: H1, D1, M1, M4 (TIER_FAST = flash-lite, which does not support thinking_config).

## M3+M4 Parallelisation

M3 (risk assessment) and M4 (triage handoff) now run in parallel. The tier-injection dependency was resolved by removing `risk_assessment` as an M4 input parameter and injecting `tier`, `action`, `rationale`, and `authorization_required_by` deterministically from the rule engine + M3 in Python after both calls complete.

Wall time for M3+M4: 13.9s. Compare: run_004 sequential M3+M4 = 21.4s (12.0+9.4). Improvement: 7.5s. Compare to run_002 sequential M3+M4 = 13.3s (8.1+5.2). Similar total — the parallelism benefit over run_002 is modest because both tasks now take ~7s each on flash (neither is as short as run_002's M4 at 5.2s).

The structural benefit is that M-stage wall time is now bounded by max(M3, M4) rather than M3+M4. As model latency improves, this will compound.
